# Pipeline Review — implementation-spec-writer — 2026-06-05

## Summary

`implementation-spec-writer` is the Stage 3c skill of the `/new-project` pipeline: it translates an approved `architecture.md` plus a `repo-snapshot.md` into a line-level, file-by-file build spec that `project-implementer` (Stage 4) executes mechanically. Its design center is correct and earns its place — it is the seam where design intent becomes deterministic build instructions, and it holds the right hard rule (DR-7-adjacent): "if the implementer has to make a judgment call, the spec is incomplete," with architecture gaps flagged-and-stopped rather than silently filled (OP-3, AP-2). The shape is sound; the prose is constraint-led where it matters and procedural where sequencing demands it. The load-bearing risk it carries today is **stale internal references to a template path that no longer exists** — the skill twice instructs the agent to read `templates/project-baseline/` with a `{{PLACEHOLDER}}` / `settings-additions.json` / `claude-md-additions.md` module structure that the repo never shipped under that name; the actual template is `templates/project-claude-md/` (constant fragments) plus `project-settings.json.template`. That is a Substantive contract drift against a tracked sibling resource. Currency against the pinned Anthropic Skills doc passes cleanly. Secondary risk is leanness: the skill carries a Step 6 "Review with User" gate that is dead under the pipeline's autonomy posture, and embeds full template prose that would read leaner as section-purpose specs.

## Innovations proposed

- **Add a context-pack input to the read set** — Step 1 reads architecture + snapshot + decisions + plan, but `/new-project` now copies `context-pack.md` into `pipeline/` as a discovered input (new-project.md line 111). The spec writer never reads it, so original mandate/scope intent is invisible at the spec seam — the exact gap `/contract-check` exists to close at the command level. Citing it would let the writer trace operations back to stated requirements. Affects: `skills/implementation-spec-writer/SKILL.md` (Input Expectation, Step 1). Risk class: minor (additive read, no shared-state effect).
- **Encode the path-resolution rule as a constraint, not a guess-ban** — Step 2 says "do not guess paths — derive them from existing patterns," which states the prohibition but not the resolution method. A constraint ("every path must match a pattern present in repo-snapshot.md or an existing sibling; otherwise flag as a gap") generalizes better per OP-6 / AP-3 (transfer the mental model, not the directive). Affects: SKILL.md Step 2 + Quality Criteria. Risk class: none.
- **Make the architecture-gap flag a structured handoff field** — gaps are flagged in prose ("Architecture gap: …"). Stage 3c's agent (`pipeline-stage-3c.md`) must then route them to the user and record in `decisions.md`. A fixed gap-record shape (component / missing-detail / options) would make the 3c→4 handoff machine-checkable rather than prose-parsed. Affects: SKILL.md § 5 (Known Limitations), Step 4. Risk class: minor.

## Leanness fixes

- **Cut Step 6 "Review with User"** — the three review questions ("Does the operation order make sense?" etc.) are a rubber-stamp gate (AP-4) that contradicts the workspace decision-point posture and the pipeline's autonomy model. Stage 3c approval is owned by the orchestrator's gate protocol and the upstream Architecture Gate, not by an in-skill Q&A. Estimated token reduction: small (~120 tokens). Risk class: minor (behavioral — removes a gate the harness already supersedes; QC the cut).
- **Replace embedded template prose with section-purpose specs** — the per-component operation templates (NEW SKILL / AGENT / COMMAND blocks, lines 71–199) are large literal scaffolds. Much of their content restates the canonical component shapes that already live in `templates/` and the creation pipeline. Pointing to the canonical fragment for the constant parts would shrink the skill. Estimated token reduction: medium (~300–500 tokens) — verify against QS-6 before trimming, as some detail is load-bearing for "mechanical" specs. Risk class: minor.
- **Collapse duplicated baseline-integration instructions** — baseline-template handling appears twice (Input Expectation "Project baseline template" para + Step 4 "Baseline template integration"). One canonical statement plus a back-reference is leaner. (Note: both copies are currently wrong — see Brokenness; fix path and de-duplicate in one edit.) Estimated token reduction: small. Risk class: minor.

## Brokenness / drift noted

- **Stale template path `templates/project-baseline/` — does not exist** — Evidence: SKILL.md line 52 ("references baseline components or modules from `templates/project-baseline/`") and lines 255–260 ("`templates/project-baseline/`", "modules with `settings-additions.json`", "modules with `claude-md-additions.md`"). `find . -type d -name project-baseline` returns nothing; `ls templates/` shows only `project-claude-md/` (fragments: header.md, commit-rules.md, compaction.md, input-file-handling.md, session-boundaries.md), `project-settings.json.template`, `incident-log-template.md`, and `README.md`. The module structure the skill describes (`{{PLACEHOLDER}}` resolution, `settings-additions.json`, `claude-md-additions.md`) was never shipped under this path. `templates/README.md` confirms the real convention: only `header.md` carries `{{NAME}}`/`{{PROJECT_DESCRIPTION}}`; the four CLAUDE.md fragments are constants with **no** substitution tokens. The skill's "resolve all `{{PLACEHOLDER}}` markers" instruction therefore points at a contract that does not exist. Severity: **Substantive** — when an architecture references baseline components, the spec writer is instructed to read a missing path and resolve markers that aren't there; the operation block will either fail the read or fabricate a structure. This drift entered when `/new-project` was restructured (commits `553f440` shipped `project-baseline`; `692531c`/`890429b` reworked the pipeline and template layout; the skill's references were not migrated).
- **Input-provenance wording is slightly behind the orchestrator** — Evidence: SKILL.md lines 47–48 describe `technical-spec.md` and `project-plan.md` as "copied into `pipeline/` by `/new-project` from the project-planning workspace" — correct — but the skill omits `context-pack.md`, which new-project.md line 111 now also copies into `pipeline/`. Severity: Minor (incompleteness, not contradiction; folds into the Innovation above).
- **Currency check — passed.** Evidence: WebFetch of `https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview` (2026-06-05). Current platform convention requires only `name` (≤64 chars, lowercase/numbers/hyphens) and `description` (non-empty, ≤1024 chars); both are present and well-formed in the skill's frontmatter. `model: sonnet` and `effort: medium` are workspace-canonical tier extensions (permitted per Model Tier rule, not docs-mandated) and introduce no divergence. No frontmatter or structural drift against current Anthropic Skills guidance.
