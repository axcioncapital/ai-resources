# Pipeline Review — implementation-project-planner — 2026-06-05

## Summary

`implementation-project-planner` is the process-knowledge skill behind `/plan-draft` in the `projects/project-planning/` workspace; it turns an approved context pack into a Claude Code implementation plan and, via the CR-15 harness-ready field guidance, supplies tier/dependency/session-packaging process knowledge for project plans of *any* type. Its design center is correct: it is a process-knowledge skill, not a template, with the structural authority living in `ref-project-plan.md` and the skill subordinated to it by `/plan-draft` Step 3. The shape is sound, but the skill carries two load-bearing drift risks: its own "Implementation Project Plan Structure" (Sections 1–8) is a second, competing schema for the same artifact that `ref-project-plan.md` already owns (Elements 1–8), and the skill body nowhere states the subordination that `/plan-draft` relies on. There is also a scope contradiction between the `description` (research planning is out of scope) and Step 3 (research planning is explicitly in scope when invoked via `/plan-draft`). Currency against the pinned Anthropic agent-skills doc is clean. No brokenness blocks the pipeline today; the dual-schema coupling is the one Substantive item.

## Innovations proposed

- **State the schema subordination inside SKILL.md** — `/plan-draft` Step 3 declares `ref-project-plan.md` authoritative for structure and this skill process-only, but the skill body asserts "Every plan contains these sections in order" (line 52) with no pointer to the reference. A fresh executor reading the skill alone would treat the skill's 8 sections as canonical. One line at the top of "Implementation Project Plan Structure" naming `ref-project-plan.md` as the structural source of truth removes the silent conflict (principles.md § OP-3 — loud over silent; AP-1 — silent conflict resolution). Affects: `skills/implementation-project-planner/SKILL.md`. Risk class: minor (canonical skill edit — DR-8 `/risk-check` applies before commit per risk-topology.md § 3).
- **Resolve the research-scope contradiction in the description** — the `description` says "Do NOT use for research project planning (use task-plan-creator)" yet Step 3 and the worked example serve a three-unit *research* project via `/plan-draft`. The two statements disagree on a load-bearing routing boundary (principles.md § AP-1). Rewrite the description to scope the exclusion to *standalone* invocation only, mirroring the Failure Behavior block (line 249) which already draws that line correctly. Affects: `skills/implementation-project-planner/SKILL.md` frontmatter. Risk class: minor.
- **Cross-link the autonomy-tier definitions to a single source** — the A/B/C/D tier rules are stated in full in three places: this skill (Step 3, lines 177–181), `ref-project-plan.md` (element 6e), and they reference workspace Autonomy Rules. Three verbatim copies of a load-bearing rule is the DR-7 / AP-7 drift surface — a change to one tier definition silently diverges the others. Designate `ref-project-plan.md` § 6e as the definition and have the skill cite it rather than restate it. Affects: `skills/implementation-project-planner/SKILL.md`, `projects/project-planning/pipeline/ref-project-plan.md`. Risk class: minor.

## Cross-resource interactions

- **`/plan-draft` (project-planning command) — composition gap (latent)** — `/plan-draft` Step 3 declares this skill subordinate to `ref-project-plan.md`, but the skill does not echo that contract. The hand-off works today because `/plan-draft` reads the reference *first* (Step 2) and the skill *second* (Step 3) as guidance. The gap is latent: anyone invoking the skill standalone, or editing the skill's Section 1–8 block believing it authoritative, breaks the implied contract. Risk: advisory (the ordering in `/plan-draft` holds the contract together; tighten via Innovation 1).
- **`context-pack-builder` — upstream contract (clean)** — the skill consumes the approved context pack's element 9 authority tags and maps them directly to the Canonical Inputs ranking (`primary-authority` → must-read, etc., lines 173–174). `context-pack-builder` produces those tags upstream. The mapping is stated identically in both the skill and `/plan-draft` line 21 — consistent, no drift. Risk: none.
- **`implementation-spec-writer` — downstream, no direct coupling** — the skill explicitly disclaims spec writing (Stage 3c) and architecture design (Stage 3b). The hand-off is mediated entirely through the `project-plan.md` artifact and the `/new-project` orchestrator (Stage 3a→3b reads `project-plan.md`, per `pipeline-stage-3b.md` line 18); these two skills never call each other. The boundary is clean and the exclusion list in both descriptions is mutually consistent. Risk: none.
- **`new-project.md` / `pipeline-stage-3b` — downstream consumer of the produced plan** — `/new-project` discovers `project-plan-v{N}.md` from `projects/project-planning/output/{name}/` and copies it into the pipeline dir for Stage 3b (`new-project.md` lines 30, 60, 158). The contract is the *artifact*, governed by `ref-project-plan.md`, not this skill — so any plan-schema change must be made in the reference (the single authority), which reinforces Innovation 1 and the dual-schema Brokenness item. Risk: advisory (a schema edit made only in the skill would not reach `/new-project`'s expectations, which track the reference).
