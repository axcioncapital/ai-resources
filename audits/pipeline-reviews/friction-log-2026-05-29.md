# Pipeline Review — friction-log — 2026-05-29

## Summary

`/friction-log` is a 42-line two-mode logger — `start` initializes a session block; bare-args appends a timestamped entry with cheap context capture (commit short-hash + session-notes title) and a stub-check. The design center is correct for what it does: zero-prompt write, no questions asked, append-only. The shape risk it carries today is not in its own body — it is in the **three-writer contract** it shares with `/note` (sibling) and `friction-log-auto.sh` (PreToolUse hook). Two of the three writers emit `## Session — {date}` (em-dash, two-hash). The hook emits `### Session: {date} — Trigger: /name` (colon, three-hash). The contract note in `note.md` line 16 explicitly asserts "byte-identical to what `/friction-log` writes" and "the three writers stay mutually detection-compatible" — the first claim is false, the second holds only because detection keys on `### Friction Events`, not on the session header. Per `risk-topology.md § 5 Signals that elevate a change to structural risk`, a string-literal contract between components is a two-end contract; this one has drifted silently. The currency-check against the unified Anthropic skills/commands doc surfaces additional minor frontmatter gaps — described below. The operator override that kept this command on weekly tier is, on this read, **over-tiered**: the command itself is single-step and stable; the contract risk lives in the hook + sibling command, not in the body. Quarterly placement would do the same work at one-third the cycle cost.

## Innovations proposed

- **Unify the session-header shape across all three writers** — Per `principles.md § OP-3` (loud over silent) and `risk-topology.md § 5` (string-literal two-end contracts), the right fix is one canonical header shape, written by one canonical helper or one canonical string constant. The cheapest landing is to rewrite `friction-log-auto.sh` lines 41–46 so the appended block matches `## Session — {YYYY-MM-DD HH:MM}` + `### Friction Events` + `#### Write Activity` exactly. Affects: `.claude/hooks/friction-log-auto.sh`, with downstream verification in `.claude/commands/friction-log.md` Step 2 and `.claude/commands/note.md` Step A.2. Risk class: structural change class (hook edit + downstream contract). `/risk-check` required per `risk-topology.md § 3`.

- **Promote `friction-log: true` frontmatter to a documented convention** — Today only `cleanup-worktree.md` declares it. The hook is silently dormant for every other command. Per `principles.md § OP-6` (transfer mental model, not just instructions), document this flag in `docs/session-guardrails.md` or a new short doc so future command authors know it exists and what it triggers. Affects: `docs/session-guardrails.md` (or a new file). Risk class: minor (docs only).

Per `principles.md § DR-7` (generalize only when second consumer exists — and here both consumers exist), extract the canonical block to a single source. ## Brokenness / drift noted

- **Three-writer contract drift — Substantive.** `friction-log-auto.sh:42–46` writes `### Session: $NOW — Trigger: /$SKILL_NAME` (three-hash, colon). `friction-log.md:21` writes `## Session — {YYYY-MM-DD HH:MM}` (two-hash, em-dash). `note.md:18` writes the same as `friction-log.md`. Live log (`logs/friction-log.md:3,12,21,35,…`) shows the `## Session —` shape in every entry — the hook has not been observed firing into the log (or its entries are absent from the recent tail). The contract assertion in `note.md:16` ("byte-identical to what `/friction-log` writes" + "mutually detection-compatible") is false on the first claim and is held together only by the second-level invariant that detection keys on `### Friction Events`, not on the session header. Severity: Substantive — the moment any consumer (e.g., `friday-checkup.md:313` friction-log-dormancy scan; `open-items.md:35` friction-log entry parser) keys on the session-header pattern, the hook-written entries become invisible.

- **Path shape uses leading-slash `/logs/friction-log.md` — Minor.** Lines 19 and 32 reference `/logs/friction-log.md` with a leading slash, which on a literal filesystem read resolves to the OS root, not the project root. Every other consumer in the repo (`friday-checkup.md:139,235,236,313`; `improve.md:9,15`; `repo-dd.md:165`; `wrap-session.md:226`; `monday-prep.md:165`; `deploy-workflow.md:126`; `prime.md:85`; `friday-act.md:79–80`) writes `logs/friction-log.md` or `ai-resources/logs/friction-log.md` — relative or workspace-rooted. The body has always worked in practice because Sonnet interprets `/logs/...` as "the logs/ at the project root" — but that is inference, not the literal contract. Severity: Minor (cosmetic; consistent across the file). Fix: drop the leading slash in steps 2 and 3.

- **Currency check against `https://code.claude.com/docs/en/skills` — Minor.** The unified skills/commands doc (read 2026-05-29) documents `description`, `argument-hint`, `allowed-tools`, and `disable-model-invocation` as canonical frontmatter fields for commands and skills. `friction-log.md` frontmatter declares only `model: sonnet`. Missing fields: `description` (recommended per the doc — "what the skill does and when to use it"), `argument-hint` (would document the `start | [description]` shape for autocomplete), and `disable-model-invocation: true` (this is a side-effect-bearing log writer; the doc names `/commit`, `/deploy`, `/send-slack-message` as the canonical class — `/friction-log` belongs in that class). Severity: Minor — cosmetic against current platform guidance; functional today. Per `principles.md § QS-6` (command completeness), the missing `description` field is the gap that matters most: it costs no tokens until autocomplete is queried, and it tells future-Claude what the command is for.

- **Stub-check rule duplication with `/note` — Minor.** The stub-check rule (under ~15 chars OR placeholder-pattern match) is restated verbatim in `friction-log.md:40` and `note.md:31`. If the threshold or pattern ever changes, both files must be updated in lockstep — silent drift class. Severity: Minor. Fix: extract to `docs/friction-log-discipline.md` (or similar) once a second drift signal appears. Defer per `principles.md § DR-7`.

## Cross-resource interactions

- **`/note` (sibling command) — contract drift** — `note.md` Step A is functionally a superset of `/friction-log` append-entry mode: it writes the same session block, the same entry shape, the same stub-check, into the same file. Two writers, one log. Today's invariant: the operator types `/note friction: …` or `/friction-log …` interchangeably and gets the same row. Risk: Substantive — if either body diverges (e.g., a fix to one's stub-check is not propagated), the log silently bifurcates. The two should converge on one canonical writer (`/note friction:` calls `/friction-log` internally, or the append-entry logic lives in a shared doc both pages reference).

- **`friction-log-auto.sh` (PreToolUse hook) — contract drift** — Already named above under Brokenness. The hook is the third writer and the one that has drifted hardest from the canonical shape. Risk: Substantive.

- **`/friday-checkup` Step 313 (friction-log-dormancy scan) — downstream consumer, advisory** — Reads the most recent `### YYYY-MM-DD` or `## YYYY-MM-DD` header. The friction log uses `## Session — YYYY-MM-DD HH:MM` (two-hash + word + date). The regex in the dormancy scan matches the date prefix but not the `## Session —` literal — meaning any markdown that has a stray `## 2026-...` header anywhere in the file will be read as the freshest entry. Risk: advisory — practical-impact is low because the file is well-disciplined, but the parser shape is loose. Flag for awareness; not a load-bearing fix for this cycle.

- **`/open-items` (logs/friction-log.md row, line 35) — downstream consumer, advisory** — The entry-parser model assumes one canonical entry shape (top-level `-` bullet under `### Friction Events`). Both `/friction-log` and `/note` produce this shape. The hook produces it too. Today's invariant holds. Risk: advisory — same drift class as `/friday-checkup` above; not the consumer to fix first.

- **Registry tier signal — operator override revisited** — The registry (`audits/pipeline-review-registry.md` lines 26 + 70 + 76) records: weekly tier kept by operator override; System Owner had recommended quarterly. On this read, the **System Owner recommendation was right**. The command's own body is single-step, stable, and rarely changes (last touched 2026-05-22, two commits ago). The risk that justifies weekly attention lives outside this file — in the hook contract and the sibling-command duplication. Reviewing `/friction-log` weekly will not surface those issues faster, because they are not in the file under review. Recommendation: re-tier to quarterly. Risk: none (the only cost of being wrong is one extra cycle every ~10 weeks). Per `principles.md § OP-9` (the agentic-era driver does not license speculation): weekly cadence on a stable command is the convenience-feature shape that OP-9 cautions against.
