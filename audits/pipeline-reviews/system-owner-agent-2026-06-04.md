# Pipeline Review — system-owner (agent definition) — 2026-06-04

## Summary

The `system-owner` agent is the judgment substrate behind seven advisory functions (A–G) routed by `/consult`, `/architecture-review`, `/implementation-triage`, `/systems-review`, `/friday-so`, and `/so-monthly`. Its design center is a clean separation between three always-read voice/grounding references (persona, grounding, toolkit-relationship), a per-function read map that bounds context, and a write-scope hard-fenced to `projects/axcion-ai-system-owner/output/`. The contract is well-shaped: declarative co-ownership voice, decline-when-ungrounded as a hard rule, and a ≤30-line return contract that conforms to `ai-resources/CLAUDE.md § Subagent Contracts`. The shape risk it carries today is grounding drift, not design drift — the references were recently reconstructed, and the agent's own Phase 1 read set (`toolkit-relationship.md § 4`) still points at the deprecated `output/phase-1/` baseline while `grounding.md § 1` declares the vault canonical. The agent file is also not git-tracked, so this canonical judgment component exists only as working-tree state. Both are Substantive drift, not Blocking — the vault paths the agent actually reads at runtime all resolve correctly.

## Innovations proposed

- **Collapse three always-read references into a load-order note** — persona, grounding, and toolkit-relationship are read on every invocation regardless of function. The toolkit-relationship read (`~2.5K`) is only load-bearing for Function B (change-shape). Gating it behind the change-shape classifier rather than the always-on Phase 1 set would cut per-invocation cost on the five non-B functions. Rationale: OP-1 (compound value) favors token efficiency on the hot path; the agent is invoked from six commands. Affects: `ai-resources/.claude/agents/system-owner.md` (Phase 1), `grounding.md § 2`. Risk class: structural change class (canonical-agent edit per `risk-topology.md § 3`) — requires `/risk-check`.
- **Add an explicit grounding-version assertion to Phase 1** — the agent should read `grounding.md § 1`'s "Current canonical (v2): vault/" line and refuse to read any `output/phase-1/` path. This would make the deprecated-baseline drift in `toolkit-relationship.md § 4` inert at runtime even before that file is corrected. Rationale: OP-3 (loud over silent) — a stale path read silently degrades grounding without operator signal. Affects: `ai-resources/.claude/agents/system-owner.md` (Phase 3). Risk class: minor.

## Leanness fixes

This is duplicated instruction across two always-loaded surfaces (the agent body and `persona.md`). A one-line pointer ("apply `persona.md § 5` voice rules 1–7") preserves the contract without the restatement. Rationale: DR-5 (cross-session content earns its place) applied to agent bodies; AP-3 (constraints over duplicated procedure). Estimated token reduction: small (~300–400 tokens per invocation). Risk class: minor — but the restatement may be deliberate redundancy for an agent that could be invoked with persona.md unread; verify the persona read is truly unconditional before trimming. `[CITATION NEEDED]` on whether agent-body redundancy of an always-read reference is sanctioned anywhere in the grounding base.

Functions A and B already share one output contract by cross-reference; C/E/F/G each define a full section structure that overlaps in shape (header + summary + findings). No action proposed beyond noting the surface — these are genuinely distinct contracts and AP-7 (no speculative abstraction) cautions against forcing a shared template before a second consumer needs it. Estimated token reduction: none recommended. Risk class: none.
