# Pipeline Review — new-project — 2026-06-05

## Summary

`/new-project` is the workspace's largest orchestrator: a Sonnet dispatch command that discovers approved planning artifacts from `projects/project-planning/output/{name}/`, scaffolds the project shell, and drives a six-stage gated pipeline (3a Snapshot → 3b Architecture → 3c Spec → 4 Implementation → 5 Testing → 6 Session Guide) via six stage agents. Its design center is correct: the orchestrator holds no judgment — it sequences gates, persists `pipeline-state.md` as the single source of truth, and delegates every analytical stage to a tier-appropriate agent (3b/3c on Opus, the rest on Sonnet), which is exactly the "automate execution; gate judgment" split (principles.md § OP-2) the system mandates. Shape risk today is concentrated not in the stage logic but in the long bash-heavy Post-Pipeline Enrichment block, which carries the highest line count, the most two-end contracts, and one rule-drift defect. The command is load-bearing per risk-topology.md § 2 (affects buy-side-service-plan, corporate-identity, project-planning) and is correctly a Critical/weekly registry entry. This is its first deep review — the registry shows `Last reviewed: never`. The pipeline is structurally sound and current against Anthropic platform docs; the findings below are one Substantive rule-drift fix plus targeted leanness.

## Innovations proposed

- **Promote enrichment to a `pipeline-stage-6.5` agent** — Post-Pipeline Enrichment is ~420 of the command's ~666 lines and runs five sequential bash blocks (settings merge, additionalDirectories grant, CLAUDE.md fragments, logs scaffold, git setup). It is execution work with no operator judgment between blocks, so it belongs in a delegated Sonnet agent like every other stage — keeping it inline bloats the orchestrator's always-loaded body and forces re-read cascades (see usage-log evidence, Leanness below). This matches the orchestrator-holds-no-execution-logic pattern the six stage agents already establish. Affects: `.claude/commands/new-project.md`, new `.claude/agents/pipeline-stage-enrichment.md`. Risk class: structural change class (new canonical agent + canonical command edit) — gated by `/risk-check` per risk-topology.md § 3.
- **Add a smoke-test gate after enrichment** — usage-log line 436 records ~2–4k tokens/session lost to a "smoke-test gap" on new-project script provisioning (~1-in-5 sessions). A bounded post-enrichment check (does `auto-sync` symlink the 10 canonical commands; does `settings.json` parse; does `additionalDirectories` resolve) closes the gap the log already names. This is closure-channel work, which the system prioritizes (principles.md § OP-12, closure before detection). Affects: `.claude/commands/new-project.md` (extend Step 5a's verification posture to cover settings + grant). Risk class: minor (additive verification, no behavior change to existing steps).
- **Lift the model-profile identifiers into the settings template** — Step 11a hardcodes `claude-sonnet-4-6[1m]` and `claude-opus-4-7` inline and then verifies them against `buy-side-service-plan/CLAUDE.md` at runtime. The verification-against-precedent is sound, but the canonical strings should live in `templates/` alongside the settings/CLAUDE.md fragments already centralized there (single-source-of-truth, the same move commits `39b27b5` already made for steps 2+4). Affects: `.claude/commands/new-project.md` Step 11a, `templates/`. Risk class: minor.

## Leanness fixes

- **Extract the enrichment bash blocks to an agent (token reduction overlaps Innovation 1)** — the orchestrator body loads in full on every `/new-project` turn; ~420 lines of bash that only execute once at pipeline end pay a per-turn cost across all six stages. Moving them to a delegated agent removes them from the orchestrator's resident context. Estimated token reduction: large (~6–8k off the always-resident orchestrator body). Risk class: structural (tracks Innovation 1's `/risk-check`).
- **Collapse the duplicated walk-up-to-`ai-resources/` idiom** — the `while [ "$d" != "/" ]` walk-up appears five times (Step 2 line 2, First Run step 2, enrichment steps 2/3/4). usage-log lines 308–315 record this exact file as the canonical re-read-cascade offender ("5+ reads on a 698-line file"). A single documented helper paragraph the blocks reference, rather than five inlined copies, shrinks the surface and the re-read pressure. Estimated token reduction: small-to-moderate (~1–2k). Risk class: minor (the idiom is identical across copies; mechanical consolidation).
- **Prune the four-section verdict-schema commentary in First Run step 4** — the ~15-line comment block explaining the `plan-qc-verdict.md` four-section schema notes the schema "has no current downstream consumer" for the spec form. Load-bearing contract notes earn their place; commentary on a non-consumed schema is prose padding that belongs in the producer command, not here. Estimated token reduction: small. Risk class: minor (the regex and its two-end contract warning stay; only the non-consumed-schema commentary trims).

## Brokenness / drift noted

- **Step 5b pushes autonomously, contradicting the current gated-push rule** — Evidence: `new-project.md` line 652 — `**Push:** ... push the initial commit (git -C projects/{name} push -u origin main)`. The git log confirms the regression vector: commit `fa51d00 batch: remove git-push gate — push proceeds autonomously after commit`. That rule was inverted on 2026-05-29 — workspace CLAUDE.md (`## Push behavior`), ai-resources CLAUDE.md (`## Commit Rules`), and user memory (`feedback_push_gated`) now all require pushes to be gated, batched to session end, and confirmed via a single operator prompt. Step 5b violates all three. This is contract drift between the pipeline and the canonical workspace rule (principles.md § OP-3 — silent divergence between practice and written rule must be corrected). Severity: Substantive.
- **Currency check passed** — Evidence: WebFetch of `https://code.claude.com/docs/en/skills` (the pinned `command` URL). Current docs confirm custom commands are merged into skills and `.claude/commands/*.md` files keep working with the same frontmatter reference. The command declares `model: sonnet` only; per the fetched frontmatter table, `name` and `description` are both optional (`No` / `Recommended`) for command-shaped files, and `model` is a documented field accepting `/model` values. No frontmatter divergence. Severity: clean.
- **No unresolved dependency gaps** — Evidence: all six pre-flight stage agents resolve (`pipeline-stage-3a/3b/3c/4/5.md`, `session-guide-generator.md`); the four load-bearing templates (`templates/project-settings.json.template`, `templates/project-claude-md/{header,input-file-handling,commit-rules,compaction,session-boundaries}.md`), the `implementation-triage.md` gate command, and the `auto-sync-shared.sh` hook all exist on disk. Severity: clean.
- **Stale model attribution in commit heredocs (cosmetic)** — Evidence: `new-project.md` lines 626 and 642 hardcode `Co-Authored-By: Claude Sonnet 4.6 (1M context)` in the generated commit messages. This pins an attribution string that will drift as the operating model changes and does not match the workspace commit-footer convention. Severity: Minor.

## Cross-resource interactions

- **`/implementation-triage` — composition (invoked at the 3b→3c gate)** — `new-project.md` Step "Stage 3b → 3c Architecture Gate" dispatches `/implementation-triage` via the Skill tool and parses its first-line verdict (`WORTH-DOING` / `MARGINAL` / `NOT-WORTH-DOING` / `DECLINE`). This is a real two-end contract on the verdict token set. The gate already handles the failure path correctly (treats triage error or unparseable line as non-blocking). Risk: advisory — any change to `/implementation-triage`'s verdict vocabulary silently breaks the parse; worth a two-end-contract note in both files. `/implementation-triage` is a quarterly registry row (`Last reviewed: never`), so it is on a slower cadence than its consumer.
- **`/plan-evaluate` + `/spec-evaluate` — contract drift surface (verdict regex)** — First Run step 4 greps `^\*\*Verdict:\*\*\s+\**PASS` against `plan-qc-verdict.md` and `spec-qc-verdict.md`, produced upstream in `projects/project-planning/`. The command body already documents this two-end contract and names both producers (commits `881f37e`, `a5b0aaa`). Neither producer is in the pipeline-review registry. Risk: advisory — the contract is documented and the check is advisory-only (warns, never aborts), so drift degrades gracefully.
- **`auto-sync-shared.sh` (SessionStart hook) — downstream dependency** — enrichment steps 2, 3, and 5 wire and invoke this hook; it is a Critical load-bearing component per risk-topology.md § 1 ("All project sessions lose access to shared resources" if broken). `/new-project` is the component that installs the per-project hook entry, so a change to the hook's interface ripples here. Risk: advisory — the wiring is template-sourced, which localizes the blast radius.
