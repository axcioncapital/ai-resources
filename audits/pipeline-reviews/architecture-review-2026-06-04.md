# Pipeline Review — architecture-review — 2026-06-04

## Summary

`/architecture-review` is a thin orchestrator: it locates the most-recent on-disk audit outputs (`/audit-repo`, `/token-audit`, optional `/repo-dd` and `/analyze-workflow`), reads them, and hands the bundle to the `system-owner` agent under Function C for a severity-grouped architecture-health synthesis written to `output/architecture-reviews/`. The design center is correct and well-defended — it reads existing outputs rather than re-invoking the audits (locked Decision 4), it pushes the heavy synthesis to an Opus subagent that owns the grounding read map, and it applies a degraded-mode posture when an audit is missing or stale. The shape risk it carries today is narrow but real: a freshness/cross-resource contract gap between this command's hard-coded glob patterns and the actual filenames its sibling audit commands write — if those drift, this command silently degrades to MISSING without anyone noticing, which is a quiet violation of OP-3 (loud failure over silent continuation). Frontmatter and structural conventions are current against the unified Anthropic skills/commands doc. The command is lean; no padding worth cutting.

## Innovations proposed

- **Glob-pattern contract assertion against sibling write paths** — Step 2's four glob patterns are an unverified two-end contract with the audit commands that produce the files. `risk-topology.md § 5` flags exactly this ("Change modifies a string literal matched by another component — two-end contract") as a signal that elevates a change to structural risk. The command should, at minimum, name in its executor notes which command writes each glob target, so a future audit-filename rename surfaces the break. Affects: `ai-resources/.claude/commands/architecture-review.md` (Step 2 table), and by reference the write paths in `audit-repo.md` / `token-audit.md` / `repo-dd.md`. Risk class: structural-adjacent — documentation/assertion only, no behavior change, so `minor` per audit-discipline.md.
- **Surface degraded-mode in the report body, not only the chat tail** — degraded-mode notes are passed into the agent brief (Step 4) and also echoed as a one-line chat warning (Step 5), but the persisted report's value as a historical record is weakened if a reader opens the file later and the MISSING/STALE provenance is buried. The agent's Function C output contract already specifies "degraded-mode markers if any audit was missing or stale" in the Header — so this is satisfied at the agent layer; the innovation is to make Step 4's brief explicitly require the markers in the *written header*, closing the gap between command intent and agent output. Affects: `architecture-review.md` Step 4 brief. Risk class: `minor`.
- **Tie staleness threshold to a named source rather than a bare literal** — the 7-day threshold (Step 2, Step 4 notes) is a v1 magic number. Where this matters is consistency with the pipeline-review-registry's own ~10–14-week rotation and `/friday-checkup` cadence: a 7-day audit-freshness floor is far tighter than the cadence that actually produces the audits, so most runs will trip STALE. This is a calibration question worth a single explicit decision rather than a silent default. Affects: `architecture-review.md` Step 2/Step 4. Risk class: `minor` (single-literal tuning).

## Cross-resource interactions

- **`/audit-repo`, `/token-audit`, `/repo-dd`, `/analyze-workflow` — upstream/downstream (producer→consumer)** — these four are the upstream producers whose on-disk outputs this command consumes by glob. The contract is filename-shaped and one-directional. Risk: `advisory` — if any producer renames its output file, this consumer silently degrades to MISSING. This is the same two-end-contract risk class `risk-topology.md § 5` names; worth a named assertion (see Innovations bullet 1).
- **`system-owner` agent (Functions A–G) — shared dependency** — `/architecture-review` is one of six consumers of the same agent. The Function C output contract lives in the agent body, not the command. Risk: `advisory` — a change to the agent's Function C output shape (Header/Executive Summary/Findings-by-Severity/Recommended Actions) is invisible to this command, which only references the structure indirectly via its Step 4 brief. The brief in Step 4 and the agent's Phase 5 Function C section currently agree; keep them in sync.
- **`/friday-so` — composition/handoff** — `friday-so.md` (Function F) consumes "a recent architecture-review if available (or a MISSING note)" per the agent body line 48. So this command's output is itself an input to a downstream sibling. Risk: `advisory` — the handoff is loose (MISSING-tolerant), so no contract break, but a renamed `architecture-review-{DATE}.md` output path would break `/friday-so`'s lookup the same way an audit rename breaks this command.
