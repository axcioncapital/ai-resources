# Pipeline Review — qc-pass — 2026-06-04

## Summary

`/qc-pass` is the workspace's independent quality gate: it stops the main session, packages a four-item handoff, and dispatches the `qc-reviewer` subagent under context isolation so the creator cannot grade its own work (QS-1). Its design center is the isolation boundary — the command body's entire job is to assemble a clean handoff and present the verdict verbatim, holding zero evaluation logic itself. The pipeline is load-bearing: `risk-topology.md § 1 High` names `qc-reviewer` as "used by every `/qc-pass`," and the command feeds three named projects plus the QC → Triage auto-loop. The shape risk it carries today is contract drift between this thin command and a much thicker agent — the command body describes a three/four-input handoff while the agent already declares a richer five-input contract with mechanical-mode and legacy fallbacks, and the command never names the rubric-selection or scope-derivation behaviour the agent owns. The command is otherwise clean, lean, and correctly tiered (`model: sonnet` — it dispatches, it does not judge). No brokenness blocks invocation; the issues are documentation-of-contract gaps and one frontmatter-currency divergence.

## Innovations proposed

- **Name the five-input contract and mechanical-mode hint symmetry** — the agent (`qc-reviewer.md` lines 14-25) accepts five inputs and a `mechanical-mode` hint; the command Step 2 lists four and Step 3 only emits `mechanical-mode: suggested`. Documenting the full contract in the command makes the two-end relationship legible and lets a future reader audit drift without opening the agent. Grounding: OP-6 (transfer the mental model, not just the procedure); the missing "why" is why mechanical-mode exists. Affects: `ai-resources/.claude/commands/qc-pass.md`. Risk class: minor (canonical command edit → `/risk-check` per `audit-discipline.md`, but body-prose only).
- **State the GO / REVISE / FLAG verdict vocabulary in the command** — the command tells the operator to "present the results" but never names the three possible verdicts the operator is deciding on. Surfacing the verdict set at Step 4 sets operator expectation and pairs with the QC → Triage auto-loop trigger (QS-2). Grounding: OP-6. Affects: `qc-pass.md`. Risk class: minor.
- **Cross-link the QC → Triage auto-loop entry condition** — `/qc-pass` is the upstream half of the QC → Triage loop (QS-2; CLAUDE.md "QC → Triage Auto-Loop"), but the command body does not point at it. A one-line pointer ("a REVISE verdict feeds `/triage` per QS-2") closes the loop for the reader. Affects: `qc-pass.md`. Risk class: minor.

## Leanness fixes

- **Collapse duplicate Step 4 numbering** — the body has two `4.` steps ("Present the results") and `4a.` then jumps to `5.`; the `4`/`4a`/`5` sequence reads as a numbering slip. Renumber to `4` / `4a` / `5` cleanly or fold 4a into 4. Estimated token reduction: small. Risk class: minor (mechanical substitution, mechanical-mode QC qualifies).
- **(none further — the command is already lean.)** At ~30 lines the body is at or below the floor for a dispatch command; the "Why a subagent?" preamble earns its place by encoding the isolation rationale (QS-1) rather than padding. Do not trim it.

## Brokenness / drift noted

- **Command/agent input-count contract drift** — Evidence: `qc-pass.md` Step 2 lists four handoff items; `qc-reviewer.md` lines 14-25 declares five inputs (the fifth being the optional mechanical-mode hint) plus a legacy 3-input fallback. The command and agent disagree on how many inputs the contract has. Severity: Minor (the agent's fallback absorbs the mismatch at runtime, so no break — but the drift is the exact two-end-contract class `risk-topology.md § 5` warns about).
- **Mechanical-mode hint values under-specified in command** — Evidence: `qc-pass.md` Step 3 emits only `mechanical-mode: suggested`; the agent (`qc-reviewer.md` line 22, line 37) also honours `forced-off`. The command cannot express the force-full-rubric case. Severity: Minor.
- **Currency check (subsumed from `/audit-critical-resources`, 2026-05-29).** WebFetch of `https://code.claude.com/docs/en/skills` succeeded. Current platform guidance: custom commands have been merged into skills; `.claude/commands/*.md` files "keep working" and "support the same frontmatter reference." The pinned-doc convention names `description`, `model`, `allowed-tools`, `argument-hint`, `arguments`, `disable-model-invocation`, and `disallowed-tools` as the frontmatter field set. Divergence: `qc-pass.md` frontmatter declares only `model: sonnet` — it carries no `description` and no `disable-model-invocation: true`. Under current docs, absence of `description` means Claude may auto-invoke `/qc-pass` when a conversation "looks like" it needs QC, and absence of `disable-model-invocation` leaves model-invocation enabled. For a command whose whole contract is "the operator stops and invokes it," model-triggered invocation is a functional divergence — it could fire `/qc-pass` outside an operator decision and pollute the QC → Triage loop. Severity: Substantive. Recommended remediation: add a `description` and `disable-model-invocation: true` to frontmatter (canonical-command edit → `/risk-check` gate per DR-8; this is a frontmatter change, so DR-9 top-3 analysis also applies — top-3 consumers are corporate-identity, nordic-pe-landscape-mapping, project-planning per `risk-topology.md § 2`).
- **Frontmatter incompleteness vs QS-6** — Evidence: `qc-pass.md` frontmatter (lines 1-3) has no H1 title and no stated trigger condition. QS-6 command-completeness requires "an H1 title matching its invocation name," "a declared `model:`," and "trigger conditions stated and unambiguous." Only `model:` is satisfied. Severity: Minor (this is the same gap the currency-check `description` fix would partly close; QS-6 is the internal-principle mirror of the platform-doc divergence above).

## Cross-resource interactions

- **`/triage` (triage-reviewer) — downstream composition** — `/qc-pass` is the producing half of the QC → Triage auto-loop (QS-2; `risk-topology.md § 1` names triage-reviewer "auto-spawned on every QC findings batch"). The command body does not name this hand-off, so the composition is correct in CLAUDE.md but invisible in the command. Risk: advisory.
- **`/friday-journal` Step 5.5 and `/analyze-transcript` Step 8b — shared agent consumer** — `qc-reviewer.md` line 3 declares three callers: `/qc-pass`, `/friday-journal` Step 5.5, and `/analyze-transcript` Step 8b. Any change to the qc-reviewer input contract driven by a `/qc-pass` review must be checked against the other two callers before landing — they share the agent. Risk: Substantive (a contract edit made "for qc-pass" silently re-shapes two other pipelines). Note: `/analyze-transcript` is not in the pipeline-review registry; flag for the operator whether it should be tracked.
- **`/refinement-pass` (refinement-reviewer) — sibling-command parallel** — `qc-reviewer.md` line 131 pins its 10-finding cap "in parity with `refinement-reviewer.md`'s 7-cap." The two reviewers are deliberately kept in sync; a rubric or cap change to one should be evaluated against the other. Risk: advisory.
- **Registry note** — of the other weekly-tier QC-loop entries, `/qc-pass` is the only one of the four (`qc-pass`, `triage`, `resolve`, plus the agents) reviewed this cycle; `triage.md` and `resolve.md` remain `never`. Reviewing them in the same window would let the loop be audited end-to-end rather than per-node. Risk: advisory.
