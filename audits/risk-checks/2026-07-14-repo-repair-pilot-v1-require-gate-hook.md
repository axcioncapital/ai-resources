## Dimensions

### Dimension 1: Usage Cost
**Risk:** High

- **New `PreToolUse` hook with matcher `Edit|Write` fires on every Edit/Write tool call** in any `ai-resources`-rooted session, not only on protected paths (the hook does its own internal path-matching, per the repo's own documented convention that "Claude Code's `matcher:` field only matches tool names, not paths" — `docs/permission-template.md:363`). This is the explicit High trigger in the base rubric: "adds a hook that runs per tool call (PreToolUse)."
- **`qc-reviewer.md` is one of the most frequently-spawned subagents in the workspace** — invoked by 6 commands directly and named as the independent-acceptance reviewer at every gate in at least one live project's pipeline (`projects/axcion-website/pipeline/project-plan.md` — 8 separate "6h. Independent acceptance" gates). Adding 3 new clauses (premise check, search-space validation, independent-verification) to this agent's body adds token cost to every one of those invocations, indefinitely.
- **`risk-check-reviewer.md`** is invoked ~4×/day per the plan's own citation ("346 reports, ~4 per active day"); a new Dimension 7 section adds a further fixed per-invocation cost on top of an already-heavy 6-dimension rubric.

### Dimension 2: Permissions Surface

### Dimension 3: Blast Radius
**Risk:** High

Per the Consumer Inventory above: **~130 individual file references, far in excess of the ">5 dependent callers" High bar**, driven specifically by `qc-reviewer.md` (24 symlinks + 6 direct invokers + 8 project-pipeline documents) and `audit-discipline.md` (22 direct file consumers).

### Dimension 4: Reversibility
**Risk:** High

- A plain `git revert` of the 6 file edits + the hook addition + the settings.json wiring is mechanically clean **once committed**. The risk is not post-commit revert; it is **mid-session self-lock before any commit exists to revert to.**
- **Self-lock hazard (scored explicitly, per the task's own framing — this is the correct dimension for it).** Once wired, `require-gate.sh` gates edits to **itself** (`.claude/hooks/*` is Structural tier) and to `**/settings.json` (also Structural tier, and this is exactly where the hook is wired). The plan's own "fails loud" requirement — "if python3/git is missing or the script errors → block, never pass" — means a bug introduced into the hook (including one introduced by a legitimate attempt to fix an earlier bug in it) makes it block **every subsequent Edit/Write to hooks, commands, agents, skills, and settings.json in that session**, including the edit needed to repair or unwire itself.
- **No dedicated recovery path is specified in the plan.** The only verified escape hatch this repo's actual settings make available is that the hook's matcher is `Edit|Write` only — a **Bash-tool-mediated** file write (`sed -i`, `python3 -c`, a heredoc redirect, or `git checkout -- <file>`) does not pass through the `Edit`/`Write` tool name and therefore does not trigger the hook at all (confirmed: `ai-resources/.claude/settings.json` permissions already carry a broad `Bash(*)` allow with `defaultMode: bypassPermissions`). This is a real recovery path, but it is (a) undocumented anywhere in the plan, and (b) structurally identical in shape to this repo's own most-recently-demonstrated failure pattern.
- **That failure pattern is not hypothetical — it is today's session.** Commit `900edf6` (`ai-resources`, 2026-07-14 S5, landed hours before this plan's approval): *"the guard protecting it runs on a false premise... the liveness guard false-fires on every finished session, and the only way past it is to delete the evidence it reads and retry — a working bypass, found while legitimately using the guard."* The exact "false-block → delete the guard's own evidence → retry" pattern the plan names as the thing "this repo gets burned by" is the observed behavior of a sibling guard **on the same day this plan was approved**. A bug in `require-gate.sh` is materially more consequential than a bug in the worktree-liveness probe, because it gates the entire hooks/commands/agents/skills/settings editing surface, not one destructive verb.
- **Append-only new artifact.** `logs/.gate-waiver` is a new file; a revert of the code change does not retroactively invalidate waiver lines already written to it, which is a minor but real "log mutation a revert cannot cleanly undo" component (Medium on its own, folded into the High above because it compounds the self-lock finding rather than standing alone).

## Recommended redesign

- **Fix the enforcement gap before wiring anything.** Either (a) wire `require-gate.sh` into every project's own `settings.json` using the same upward-walk idiom already used for `auto-sync-shared.sh`/`check-permission-sanity.sh` (so it fires regardless of which of the ~24 checkouts a session is rooted in), or (b) move enforcement to a mechanism that is not session-root-scoped — a git pre-commit hook, a pattern this same repo has already adopted elsewhere for a comparable reason (`logs/session-notes.md:331`: "the git pre-commit install is the binding guard; the PreToolUse registration is early-warning only"). Do not ship a gate whose own stated purpose ("regardless of which door you came through") is false for the majority of the repo's real editing sessions.
- **Name and test a self-lock recovery path before wiring the hook, not after.** Specify explicitly (in the hook's own header comment and in `docs/protected-zones.md`) that recovery from a broken `require-gate.sh` is via `git checkout`/Bash-tool file operations, never Edit/Write — and add this as an explicit test case in the plan's own §5 verification suite (a script bug that still allows Bash-tool-mediated recovery, verified by execution, not asserted).
