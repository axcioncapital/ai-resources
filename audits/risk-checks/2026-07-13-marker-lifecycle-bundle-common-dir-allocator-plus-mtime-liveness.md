## Dimensions

### Dimension 1: Usage Cost
**Risk:** Medium

- No always-loaded file (CLAUDE.md) is touched, and the allocator relocation adds no per-turn cost — it runs once per `/prime` invocation, unchanged in frequency from the existing block.
- **The new mtime-freshness heartbeat is a genuinely new per-turn automation surface.** `Stop` already fires 4 project-level hooks in `ai-resources/.claude/settings.json` (`check-stop-reminders.sh`, `coach-reminder.sh`, `improve-reminder.sh`, `auto-resolve-nudge.sh` — confirmed by direct read) plus 1 user-level hook (`afplay …Pop.aiff` — confirmed). A Stop-registered marker-touch would be a 6th hook on the single most frequently-firing event class in the workspace (every assistant turn, not once per session like `SessionStart`). If the write-up of the touch is silent (`exit 0`, no `systemMessage`), token cost is near-zero — but the *frequency class* (every turn, and per the same "every checkout" reasoning the SessionEnd hook used, almost certainly registered at USER level, i.e. machine-wide across every project) sits above the Medium calibration point ("a hook that runs once per session") without yet reaching the High bar (it is not per-tool-call, and it need not inject context). Scored Medium on frequency class, not on measured token cost — no implementation exists yet to measure.
- The plan's own Execution Step 3 does not specify whether the heartbeat write emits any operator-visible line; if it does (even a one-line diagnostic per turn), this would push to High. Worth pinning down explicitly before landing.

### Dimension 2: Permissions Surface
**Risk:** Low

- No `allow`/`ask`/`deny` change is proposed anywhere in the description, and `~/.claude/settings.json` (read directly) already runs `defaultMode: "bypassPermissions"` with `Bash(*)` / `Write(**)` at user level — every operation this change performs (writing under `.git/`, adding a hook entry, editing a script) is already within the account's existing maximal permissiveness. No settings-layer scope change, no new capability class, no `deny`-rule removal.
- **One placement nuance worth a note, not a permission risk:** writing session state under `$(git rev-parse --git-common-dir)/axcion-session-markers/` puts application state inside `.git/` — an unusual location relative to the existing convention (`logs/.session-marker*`, a normal working-tree path). This is a design-hygiene observation, not a permissions-surface finding, since no settings file needs to change to allow it.

### Dimension 3: Blast Radius
**Risk:** High

- **12 consumers found in the Step 1.5 inventory; 9 must-change.** Per the inventory, two of those nine — `concurrent-session-check.md` and `/prime`'s own Step 1a liveness read — are consumers of the exact per-id-marker liveness signal this change redefines, and neither is named in `CHANGE_DESCRIPTION`'s consumer list (which states only two of the four documented readers).
- **This is not a cosmetic omission — it breaks the fix's own stated purpose if shipped as described.** If only `detect-concurrent-session.sh` and `check-foreign-staging.sh` switch to mtime-freshness while `concurrent-session-check.md` and `/prime` Step 1a keep reading raw marker existence, the liveness contract forks: a stale-but-existing marker will read "dead" on two surfaces and "live" on the other two, in the same session, at the same moment. That directly contradicts the fix's premise (that liveness should be one consistent signal, not four independent heuristics) and reproduces the exact "detector cries wolf" defect on the two un-migrated surfaces.
- **Prerequisite consumer confirmed independently, not taken on trust:** `git worktree list` shows `ai-resources-research-workflow` checked out on `session/2026-07-13-research-workflow`; `git log --oneline main..HEAD | wc -l` = 6, `git log --oneline HEAD..main | wc -l` = 10 (branch is 10 commits behind main, 6 ahead — matches the plan's own count). `ls -la` on that worktree's `.claude/commands/prime.md` confirms a real, non-symlinked, 665-line file (vs. canonical's 680 lines) — the drift is real and the rebase/close prerequisite is correctly identified as blocking.
- **Shared infra touched machine-wide with no staging gate.** `detect-concurrent-session.sh` and `check-foreign-staging.sh` are registered at USER level by absolute path (confirmed by direct read of `~/.claude/settings.json`), so an edit to either is live in every project and every worktree the instant it is saved — the change description's own B-5 finding, independently confirmed here.
- **Contract change is not backwards-compatible during a partial rollout** — this is the specific High trigger from the framework's own calibration ("a contract change that is not backwards-compatible"): the mtime-freshness redefinition of "live" is a semantic change to an existing contract, and per the gap above it is not proposed to reach all documented readers in the same edit.

### Dimension 4: Reversibility
**Risk:** High

- `ai-resources/.claude/commands/prime.md` and `ai-resources/docs/session-marker.md` are cleanly `git revert`-able (tracked, no siblings created).
- **Execution Step 3 of the plan proposes editing `~/.claude/hooks/cleanup-session-marker.sh`** (adding a `reason` field to its diagnostic log line). `~/.claude/` is **not** a git repository (confirmed directly — this is the same finding the prior SessionEnd risk-check made and cited: *"not tracked in any git repository ... a `git revert` of this change is structurally impossible"*). Any edit there has no diff, no history, and no automated way to confirm a manual rollback restored prior state exactly.
- **Compounding factor, not the primary driver:** the two `ai-resources`-tracked hooks (`detect-concurrent-session.sh`, `check-foreign-staging.sh`) are registered user-level by absolute path with no staging step (Dimension 3 finding) — so between the moment an edit is saved and any future `git revert`, the new (and, per Dimension 5, not-yet-threshold-tuned) logic is live everywhere. If a false negative occurs during that live window and a real collision is missed, the resulting corrupted shared state (a lost update) is not something a later `git revert` of the hook script can undo — the damage, if any, already happened. This is squarely the framework's High bar ("state has propagated beyond git" in effect, even though nothing is pushed externally) and mirrors the identical concern the prior SessionEnd risk-check raised about this same subsystem.

### Dimension 5: Hidden Coupling
**Risk:** High

- **The mtime-freshness threshold is undefined in the plan, and the failure mode of an under-tuned threshold is the exact data-loss mode the whole subsystem exists to prevent.** `CHANGE_DESCRIPTION`'s own KEY QUESTIONS name this as open ("What threshold, and what happens to a session idle longer than the threshold"), and `logs/session-plan-2026-07-13-S13.md` Execution Step 3 specifies only the mechanism ("mtime freshness; refresh it on the Stop event … read a marker older than the threshold as dead") with **no numeric value and no derivation**. `Stop` fires per-turn, not continuously — a marker's mtime is only as fresh as the last turn boundary, so a genuinely live session idle longer than the threshold (long operator think-time, a long-running subagent, an operator away from the keyboard) will read as dead. If a **second** session then opens in the same checkout during that window, both `detect-concurrent-session.sh` (SessionStart) and `check-foreign-staging.sh` (PreToolUse) will report "no live foreign session" — a false negative — and the second session proceeds unaware of the first, exactly reproducing the two-sessions-one-checkout collision the entire marker protocol exists to prevent. This is an implicit dependency (a safe idle-time bound) that is neither chosen nor validated.
- **A second, independent unverified assumption: the git-common-dir mechanism's behavior across 24 different repos and its durability under git housekeeping.** Verified directly: `git rev-parse --git-common-dir` in the `ai-resources` main checkout returns the **relative** path `.git`, while the same command in the `ai-resources-research-workflow` worktree returns the **absolute** path `/Users/.../ai-resources/.git` — both resolve to the same directory, but an implementation that assumes one path *shape* (relative vs. absolute) across all 24 different consuming repos needs explicit normalization, which the plan does not mention. Separately, `CHANGE_DESCRIPTION`'s own adversarial question ("does the marker namespace then fragment" across 24 different git repos, "and is that correct or a bug") is posed but not answered in the plan — each of the 24 repos has its own common dir, so the allocation-claim channel is per-repo by construction (consistent with the existing per-repo `S{N}` namespace), but this has not been verified in any project repo other than `ai-resources` itself. Whether `git gc` / `git worktree prune` / `git clean` can remove an unrecognized directory under `.git/` was not independently testable in this sandboxed review (a write-probe to `.git/` was denied by the execution environment) and is asserted, not verified, in the plan.
- **Functional overlap is real but openly reasoned, not silent** (this keeps the dimension from being driven higher on this point alone): three mechanisms now touch the same liveness concern — the SessionEnd hook (kept as "opportunistic fast cleanup"), the new mtime heartbeat (primary signal), and `/prime`'s next-day orphan prune (documented backstop). The plan names all three roles explicitly, mirroring the reasoning the prior SessionEnd risk-check scored Medium (not High) for the same overlap pattern.
- **Undocumented-contract concern:** the mtime-freshness contract (readers must compare "now minus mtime" against a threshold, rather than testing existence) is a new implicit contract no consumer file documents yet, and — per the Dimension 3 finding — is not proposed to reach all four documented readers in the same edit.

- **OP-12 (closure before detection) — served.** The change corrects the accuracy of *existing* detection (reduces both the false-positive and, if done right, the false-negative rate) rather than adding a new scan or flag.
- **OP-5 (advisory vs. enforcement) — untouched.** No advisory mechanism gains new blocking authority beyond what already exists in `check-foreign-staging.sh`.

## Recommended redesign

- **Split the bundle.** Land the allocator relocation (git-common-dir MAX source) as its own change once it has been verified by execution in at least one project repo *other than* `ai-resources` (to close the cross-repo fragmentation question the plan itself raises but does not answer), and once the relative-vs-absolute path-shape difference observed directly in this review (`.git` in the main checkout vs. the absolute path in the worktree) is normalized in the implementation. Defer the mtime-liveness heartbeat to a separate, later change.
- **Do not ship the mtime-liveness heartbeat until two gaps close.** (1) Choose and document an explicit, derived threshold value (grounded in a measured or reasoned bound on realistic idle gaps — operator think-time, subagent duration — not left as an implementation detail), and add a verification case for a *genuinely live, long-idle* session in addition to the plan's existing "planted stale marker reads dead" test — a fix that can silently reintroduce the false-negative it is meant to remove is worse than the noise it replaces. (2) Update all four documented liveness consumers in the same edit (`detect-concurrent-session.sh`, `check-foreign-staging.sh`, `concurrent-session-check.md`, and `/prime` Step 1a itself) and the registry text in `cleanup-session-marker.sh`'s own header — not two of the four — so the liveness contract cannot fork mid-rollout. Take a timestamped backup of `~/.claude/hooks/cleanup-session-marker.sh` and `~/.claude/settings.json` before either is touched, given both are unversioned and the prior risk-check on this same subsystem already established that as a required step.
