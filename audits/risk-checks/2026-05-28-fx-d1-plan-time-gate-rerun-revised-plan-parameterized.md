## Mitigations

- **Dimension 5 (hidden coupling — runtime-override skill contract):** When editing canonical `produce-formatting.md` (Step 6), the threshold MUST be passed to the subagent task brief as an explicit named field — not relied on via implicit skill-default. Add a line to the Phase 2 subagent brief: `Mechanical trigger threshold: {threshold}` (verbatim, where `{threshold}` is interpolated from path-config). - **Dimension 5 (hidden coupling — SKILL.md unify-at-5+ + runtime-override note must be explicit and authoritative):** Step 8 changes line 62 from "Three or more" to "Five or more" and updates lines 235/294 consistently. When passed, the calling-command value is authoritative for that invocation; this skill's default applies only when no override is passed." This wording removes ambiguity about which side wins, and prevents a future subagent from interpretively splitting the difference.

- **Dimension 5 (hidden coupling — default-value semantics for the new field):** When editing `stage-5-paths.template.md` (Step 3) and `project-config-schema.md` (Step 4), add a "Default-value semantics" subsection analogous to the existing one for `Document model:` (schema.md lines 97-112). - **Dimension 4 (reversibility — silent dead-field on partial revert):** When updating nordic-pe's `reference/stage-5-paths.md` (Step 5), add the threshold value with a one-line comment naming its consumer: `**Mechanical trigger threshold:** "3+"  # consumed by produce-formatting.md Phase 0`. - **Dimension 3 (blast radius — step-ordering serialization):** Before executing the canonical produce-formatting.md edit (Step 6), the operator MUST confirm Steps 3, 4, and 5 have landed (template field, schema row, project value). Add an explicit serialization gate between Steps 5 and 6 in the session plan: "Verification gate before Step 6: confirm template + schema + nordic-pe instance all reference `Mechanical trigger threshold:` field consistently." This prevents a parser-references-undefined-field window during execution. The plan's Step 11 (STOP POINT — interim verification before swaps) already serves a similar role for the swap phase; this mitigation adds an analogous gate for the edit phase.

- **Dimension 5 (hidden coupling — DR-7 placement):** When adding the DR-7 graduation-confirmation rule to `ai-resource-creation.md` (Step 10), place it as a new numbered rule #5 in the existing rules block (lines 7-13), not as a separate section. Record the confirmation in the commit message. The 'one confirmed consumer + one non-consenting consumer' pattern is the failure mode this rule prevents." This satisfies the System Owner's recommendation while matching doc style.
