# Session Value Audit Rubric — 80/20 Review

> **When to read this file:** Read by the fresh-context `general-purpose` subagent that `/wrap-session` launches for its outcome check (canonical Step 6.4 / workspace-root Step 4.4) — it is that subagent's Session Value Audit measurement rubric. Externalized from the two `wrap-session.md` copies on 2026-07-04 (leanness refactor) so the command bodies stay lean and the rubric has one authoritative home. Not always-loaded; not read on ordinary turns.

## Byte-identical output contract (do not drift)

The subagent must emit the labeled lines below **verbatim** — same `### Session Value Audit — 80/20 Review` heading and same `TYPE:` / `VALUE:` / `SCORE:` / `GATE:` / `OPPORTUNITY:` / `DECISION:` / `LESSON:` / `RULE:` label prefixes. `/friday-checkup`'s Weekly Session Value Review greps this heading and these labels (`friday-checkup.md` Step 14.5). Changing a label or the heading silently breaks that roll-up. If you edit a label here, edit friday-checkup's extraction in lockstep.

## Stance

80/20 review — judgmental, skeptical of effort. Do **not** reward effort, length, or complexity; reward only shipped assets, changed decisions, reduced future friction, prevented recurring failures, and reusable system improvements.

## The labeled lines (emit ~10–14 lines total)

- **`TYPE: {A|B|C|D|E} — {label}`** + one-line rationale. A = High-Leverage Build (created or materially improved a reusable asset, workflow, command, client-facing artifact, strategic decision, or operating-system component). B = Necessary Maintenance (prevented degradation, resolved recurring friction, cleaned structural issues, protected important workflows from failure). C = Useful Diagnosis (clarified a problem, surfaced a blind spot, created a better decision basis — even if little shipped). D = Low-Yield Maintenance Drift (hygiene, cleanup, organization, or meta-work without clear future leverage). E = False Productivity (complexity, documentation, analysis, or process overhead that does not materially change execution, decisions, reliability, or future speed). Classify directly; hedge only on genuinely mixed evidence.
- **`VALUE: exec={H|M|L|N} decision={H|M|L|N} risk={H|M|L|N} compound={H|M|L|N} optime={H|M|L|N}`** — one line, five dimensions: exec = moved an active project forward concretely; decision = clarified what to do, stop, prioritize, approve, reject, or defer; risk = prevented likely future errors, regressions, repo confusion, workflow degradation, or wasted time; compound = created or improved a reusable command, template, rule, checklist, workflow, or knowledge asset; optime = reduced future cognitive load, manual judgment, or session time.
- **`SCORE: {X}/10`** + one sentence justifying it. Sum of five 0–2 components: output produced, decision improved, future time saved, risk reduced, reusability/compounding. **Override rule:** if the session produced no concrete output, no decision, no reusable asset, AND no risk reduction, the maximum score is 3/10 regardless of effort or session length.
- **`GATE: PASSED | FAILED | N/A`** — maintenance-justification gate. N/A unless the session was primarily maintenance, cleanup, documentation, repo hygiene, command refinement, or meta-process work. PASSED only if at least one criterion is true — name which: fixed a recurring failure / reduced future session time / prevented likely repo-or-workflow degradation / improved reliability of an important command / removed ambiguity from a high-use process / converted repeated manual judgment into a reusable rule / protected an important project from execution risk. Distinguish **asset-building maintenance** (improves future system performance — e.g., fixing a command that repeatedly causes confusion, consolidating duplicated workflows, removing a brittle manual step) from **comfort maintenance** (feels cleaner, changes nothing — e.g., reorganizing files without reducing friction, rewriting unused docs, meta-process on meta-process, polishing command language without behavior change). FAILED forces `TYPE` to D — or E if the session added complexity without value.
- **`OPPORTUNITY: Correct session | Should have been smaller | Should have been batched | Should have been manual | Should not have been done`** + one comparative sentence: the actual session vs. the better alternative.
- **`DECISION: Repeat | Repeat with constraints | Batch into maintenance | Redesign before repeating | Stop this session pattern`** + one sentence. Repeat = session type clearly creates recurring high-value outputs; Repeat with constraints = useful but expansion-prone (name the constraint: time cap, scope cap, evidence requirement, approval gate); Batch = legitimate work not worth a standalone session; Redesign = valuable goal, inefficient session format; Stop = little or no value, should not recur.
- **`LESSON: {one line}`** — one short prioritization lesson that helps the operator recognize the broader pattern.
- **`RULE: {Rule / Trigger / Why it matters / Where it should live}`** on one–two lines, OR exactly `RULE: No rule candidate.` — do NOT propose rules by default; most sessions justify none.

## Audit evidence rules

Use only what the session actually produced — the run manifest's `files_changed` (`logs/runs/{date}-{marker}.json`), the session note's `### Decisions Made` section, and the changed-files/git-log inspection are the evidence base; no separate evidence list in the output. (The note's Files Created/Modified sections were retired on 2026-07-13 by `RR-03`; the manifest replaced them as the file record. **Fallback:** in a project whose `wrap-session.md` is an un-migrated fork, no manifest is written — read the note's file blocks instead, and if neither exists, say the file evidence is unavailable rather than inferring it.) If there is no concrete evidence, say so plainly; do not inflate. The evidence guard extends to the audit: every D/E classification, FAILED gate, and non-"Correct session" opportunity call must cite a concrete observation.
