# Required Reference Files

> **When to read this file:** Before deploying this workflow template to a new project (via `/deploy-workflow` or manual setup), and when diagnosing "missing reference file" errors during workflow execution.

This is a **deployment-time contract** between the workflow template and any project that consumes it. Every research-workflow project MUST have the four reference files listed below at the named paths. If any file is missing, the canonical Stage 2–4 commands (`run-execution`, `run-cluster`, `run-sufficiency`, `run-report`) will fail at runtime — they pass these files (by path) to subagents that read them directly.

The contract is two-end and implicit in the consuming command bodies. This doc surfaces it explicitly so the requirement is visible at the template surface — not discovered by hitting a runtime error.

## The four required files

All paths below are **relative to the project root** (the directory containing the project's `CLAUDE.md`).

### 1. `reference/source-class-hierarchy.md`

- **Role:** Source taxonomy used to classify evidence by source quality and authority class (tier-1 primary, tier-2 secondary, etc.). Consumed during research execution to tag every source by class, and during analysis to weight evidence accordingly.
- **Template form in workflow:** `reference/source-class-hierarchy.template.md` — instantiated per-project with the project's taxonomy filled in.
- **Read by canonical commands:** `run-execution`, `run-cluster`, `run-sufficiency`.
- **Failure mode if missing:** Stage 2 (research execution) and Stage 3 (cluster analysis) subagents cannot apply source-class tags or weight evidence; outputs lose traceability.

### 2. `reference/quality-standards.md`

- **Role:** Chapter QC criteria — claim-permission classes, evidence-calibration rules, no-source-substitution rule, Uncertainty Disclosure and Caveat Routing rule, load-bearing tests. Single source of truth for what counts as a publishable claim.
- **Template form in workflow:** None — `quality-standards.md` is canonical to the workflow (not per-project). Projects link to or copy the canonical file from the workflow `reference/` directory.
- **Read by canonical commands:** `run-execution`, `run-cluster`, `run-sufficiency`, `run-report` (Step 4.2 a/b/c per-chapter triplet).
- **Failure mode if missing:** Stage 4 (report production) per-chapter QC subagents cannot evaluate claim permission or evidence calibration; chapter drafts ship without compliance check.

### 3. `reference/known-limits.md`

- **Role:** Project-specific scope/limit declarations consumed by `run-report` to enforce the project's bright-line boundaries (what the report does and does not cover). Also consumed by `run-sufficiency` to evaluate whether known limits explain evidence gaps before flagging them as genuine gaps.
- **Template form in workflow:** `reference/known-limits.template.md` — instantiated per-project with the project's specific scope/limit declarations.
- **Read by canonical commands:** `run-execution`, `run-cluster`, `run-sufficiency`.
- **Failure mode if missing:** Stage 3 sufficiency analysis misclassifies in-scope-but-bounded gaps as genuine evidence gaps; Stage 4 reports drift past the project's bright lines without warning.

### 4. `reference/style-guide.md`

- **Role:** Prose style enforcement for chapter output — paragraph length, italic-opener form, section-heading length, reference patterns, voice consistency. Pre-locked operator-provided artifact (not generated by Claude).
- **Template form in workflow:** None — `style-guide.md` is canonical to the workflow (not per-project). Projects link to or copy the canonical file from the workflow `reference/` directory; each project may override style choices but the file must exist.
- **Read by canonical commands:** `run-report` (Step 4.2 a/b/c per-chapter triplet, and per-chapter `review-chapter`).
- **Failure mode if missing:** Stage 4 chapter prose subagents cannot enforce style; chapter prose drifts in voice/format and review-chapter cannot flag the drift.

## Consumer fan-out (reverse map)

| Reference file | Consumed by canonical commands |
|---|---|
| `reference/source-class-hierarchy.md` | `run-execution`, `run-cluster`, `run-sufficiency` |
| `reference/quality-standards.md` | `run-execution`, `run-cluster`, `run-sufficiency`, `run-report` |
| `reference/known-limits.md` | `run-execution`, `run-cluster`, `run-sufficiency` |
| `reference/style-guide.md` | `run-report` (via the per-chapter triplet that delegates to the `evidence-to-report-writer` and `chapter-prose-reviewer` skill subagents in Step 4.2 a/b) |

## Path-passing convention (FX-C1)

The Stage 2–4 commands pass these reference files to subagents **by path, not by content** — the subagent reads each file at runtime. This is intentional per-chapter token-economy: content-passing would inject the same reference content N × 3 times across the per-chapter triplet loop. Subagents have Read access to project paths; this carve-out applies only to reference docs, not to per-chapter inputs (research extracts, cluster memos, section directives), which are passed by content per the context-isolation rule.

The path-passing convention means the file MUST exist at the named path at runtime — there is no fallback. The canonical command bodies hard-code the relative path `reference/{filename}`; renaming or relocating a file requires updating every consuming command (see "Consumer fan-out" above for the touch list).

**Runtime path resolution — each project owns its own copies.** Path resolution is **project-root-relative**, not workflow-template-relative. Subagents resolve `reference/quality-standards.md` against the project's working directory, not against the workflow template's `reference/` directory. The workflow template's own `reference/` directory (which contains canonical `quality-standards.md` and `style-guide.md`, plus `.template.md` shapes for the per-project files) is the **source** for deployment, not a runtime fallback. A deployed project MUST hold its own regular-file copies of all four files at `{project-root}/reference/{filename}` — symlinks back to the workflow template are not the operating convention (the active nordic-pe consumer holds plain file copies). At deploy time, `quality-standards.md` and `style-guide.md` are typically copied verbatim from the workflow template; `source-class-hierarchy.md` and `known-limits.md` are instantiated from their `.template.md` shapes with project-specific content.

## Other reference files (informational)

The workflow template's `reference/` directory contains additional files that are NOT part of this contract:

- `reference/file-conventions.md` — canonical to workflow; read by `audit-structure` only.
- `reference/stage-instructions.md` — canonical to workflow; read by multiple commands; describes the 5-stage pipeline.
- `reference/claim-permission.template.md`, `reference/jargon-gloss-config.template.md`, `reference/language-search-blocks.template.md`, `reference/stage-5-paths.template.md` — Stage 5 configuration templates, instantiated per-project but not in this required-files contract.
- `reference/stage-5-common-phases.md` — canonical Stage 5 phase definitions; read by Stage 5 produce-* commands.

These files have their own roles and consumer maps; they are not enumerated here because they do not participate in the four-file deployment contract. For their roles see the corresponding command bodies or `reference/stage-instructions.md`.

## Maintenance

When a canonical Stage 2–4 command is edited to read a new reference file, that file becomes part of this contract — update the table above. Conversely, when a reference file is no longer consumed by any canonical command, remove it from the table. This doc tracks the live consumer-fan-out, not a frozen historical snapshot.
